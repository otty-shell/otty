# RFC: Блоковая модель терминала в OTTY (Warp‑подобные блоки)

Автор: Ilia Shvyrialkin  
Статус: Ready
Версия: 0.1  
Дата: 01.12.2025

---

## 1. Обзор

Данное RFC описывает первую версию блоковой системы в OTTY, подобной Warp:

- Терминальная сессия представляется как последовательность **блоков**.
- Каждый блок соответствует одной команде (или одному полноэкранному приложению) и хранится в **отдельном гриде**.
- Метаданные и границы блоков передаются через специальный **DCS‑протокол с JSON** из оболочки (bash/zsh/fish).
- UI (`otty-ui/terminal`) отображает блоки визуально и предоставляет базовое API для взаимодействия (выделение/копирование блока и т.п.).

---

## 2. Цели

- Корректная разбивка истории на блоки:
  - один блок = одна команда (preexec → вывод → exit/precmd);
  - отдельный блок для полноэкранных приложений (ALT_SCREEN).
- Отдельный грид (multi‑grid) для каждого блока:
  - вывод нового блока не переписывает содержимое старых;
  - можно реализовывать операции «по блокам».
- Базовая визуализация блоков:
  - чёткое разделение блоков в UI;
  - подсветка рамкой при наведении/клике.
- API для дальнейшего развития:
  - доступ к списку блоков и их метаданным;
  - возможность расширять действия над блоками (копирование, поиск, навигация).

---

## 3. Не цели (на данную итерацию)

- Продвинутый редактор ввода команд (как Warp input editor) — вне этого RFC.
- Расширенный UX по блокам (поиск по блокам, фильтры, коллапс/группировка и т.п.) — заложены hooks и структура, но поведение минимальное.
- Сетевая/коллаборативная функциональность (шаринг сессий, CRDT) — вне области.

---

## 4. Определения

- **Блок** — логическая единица истории терминала, включающая:
  - команду;
  - вывод (stdout+stderr);
  - метаданные (id, время начала/окончания, exit_code, тип).
- **Тип блока (`BlockKind`)**:
  - `Command` — блок, начатый событием `preexec`, содержит выполнение команды.
  - `Prompt` — блок, соответствующий рендеру приглашения; для него в будущем планируются отдельные взаимодействия (ввод, редактирование и т.п.).
  - `FullScreen` — блок, соответствующий полноэкранному приложению (`ALT_SCREEN` / full‑screen TUI).
- **Командный блок** — блок с типом `Command`.
- **Промпт‑блок** — блок с типом `Prompt`.
- **Full‑screen блок** — блок с типом `FullScreen`.
- **BlockSurface** — составная поверхность, внутри которой каждый блок имеет собственный `Surface` (грид).

## 5. Протокол DCS для блоков

### 5.1. Общий формат

Блоковые события передаются через DCS‑последовательности вида:

```text
ESC P otty-block;{JSON} ESC \
```

где:

- `ESC` — `0x1B`;
- `P` — начало DCS;
- `otty-block;` — текстовый префикс, идентифицирующий протокол блоков;
- `{JSON}` — строка в формате JSON UTF‑8;
- `ESC \` (`0x1B 0x5C`) — окончание DCS.

### 5.2. Структура JSON

JSON должен иметь следующую структуру:

```json
{
  "v": 1,
  "id": "string",
  "phase": "preexec",
  "cmd": "ls -la",
  "cwd": "/home/user",
  "time": 1701340000000,
  "exit_code": 0,
  "shell": "zsh"
}
```

Обязательные поля:

- `v: u8` — версия протокола, на текущий момент `1`.
- `id: String` — уникальный идентификатор блока в рамках сессии.
- `phase: String` — стадия жизненного цикла блока:
  - `"preexec"` — начало выполнения команды;
  - `"exit"` — завершение команды;
  - `"precmd"` — начало рендера нового промпта (опциональное событие).

Опциональные поля (но поддерживаются ядром):

- `cmd` — текст команды.
- `cwd` — текущий рабочий каталог.
- `time` — timestamp в миллисекундах.
- `exit_code` — код завершения.
- `shell` — тип оболочки.

### 5.3. Семантика фаз

- `phase = "preexec"`:
  - Обозначает начало нового командного блока.
  - Ядро создаёт новый блок с новым внутренним `Surface`.
- `phase = "exit"`:
  - Обозначает завершение выполнения команды.
  - Ядро обновляет метаданные блока (`exit_code`, `finished_at`).
- `phase = "precmd"`:
  - Обозначает начало рендера нового промпта.
  - Используется как дополнительный сигнал, что предыдущий командный блок точно завершён.
  - По желанию можно использовать для создания отдельного промпт‑блока.

### 5.4. Ограничения размера

Чтобы избежать чрезмерно больших DCS‑строк, плагины для оболочек должны:

- ограничивать длину `cmd` (например, до 1024 символов, при обрезке — добавлять `…`);
- ограничивать длину `cwd` (например, до 512 символов);
- вводить максимальную длину JSON‑строки (например, 4096 байт). При превышении:
  - либо не отправлять DCS вообще,
  - либо отправлять урезанный вариант (`cmd`/`cwd` обрезаны).

Ядро OTTY (`otty-escape`) должно:

- корректно обрабатывать DCS, даже если он приходит в несколько `read()` из PTY;
- при ошибке парсинга JSON логировать и игнорировать конкретное DCS, не влияя на остальной поток.

## 6. Модель данных блоков (ядро, multi‑grid)

### 6.1. Структуры

В `otty-surface` вводится много‑гридовая модель:

```rust
struct BlockMeta {
    id: String,
    kind: BlockKind,
    cmd: Option<String>,
    cwd: Option<String>,
    started_at: Option<i64>,
    finished_at: Option<i64>,
    exit_code: Option<i32>,
    shell: Option<String>,
    is_alt_screen: bool,
}

struct Block {
    meta: BlockMeta,
    surface: Surface,
}

struct BlockSurface {
    blocks: Vec<Block>,
    active: usize,
    max_blocks: usize,
}
```

### 6.2. Лимит блоков

- `max_blocks` по умолчанию = `1000`.
- При `blocks.len() > max_blocks` самый старый завершённый блок удаляется полностью (грид + метаданные). Незавершённые блоки (например, текущая команда или активный промпт) не удаляются.

### 6.3. Обработка DCS‑событий

`otty-escape` расширяется новым действием:

```rust
enum BlockPhase {
    Preexec,
    Exit,
    Precmd,
}

struct BlockEvent {
    phase: BlockPhase,
    meta: BlockMeta,
}

enum Action {
    // ...
    BlockEvent(BlockEvent),
}
```

В `TerminalSurfaceActor` (`otty-libterm/src/terminal/surface_actor.rs`) добавляется обработка:

- `Action::BlockEvent { phase: Preexec, meta }`:
  - завершить текущий блок (если он в состоянии running);
  - создать новый `Block` с новым `Surface` и `meta` (установить `started_at`/`cmd`/`cwd`);
  - сделать его активным (`active`).
- `Action::BlockEvent { phase: Exit, meta }`:
  - найти блок по `meta.id`;
  - обновить `exit_code` и `finished_at`.
- `Action::BlockEvent { phase: Precmd, meta }`:
  - использовать как сигнал завершения предыдущего командного блока;
  - опционально — создавать отдельный промпт‑блок.

## 7. Snapshot и API для UI

### 7.1. Snapshot блоков

Для `BlockSurface` метод `snapshot_owned()` собирает единый `SnapshotOwned`:

- Все внутренние `Surface` блоков «сшиваются» вертикально в один плоский поток клеток.
- Дополнительно формируется список блоков в терминах глобальных линий:

```rust
struct BlockSnapshot {
    id: String,
    meta: BlockMetaPublic,
    start_line: i32,
    line_count: usize,
    is_alt_screen: bool,
}

struct BlockMetaPublic {
    id: String,
    kind: BlockKind,
    cmd: Option<String>,
    exit_code: Option<i32>,
    started_at: Option<i64>,
    finished_at: Option<i64>,
    shell: Option<String>,
}
```

`SnapshotOwned` расширяется полем:

```rust
pub struct SnapshotOwned {
    // ...
    pub blocks: Vec<BlockSnapshot>,
}
```

`SnapshotView` предоставляет:

```rust
impl<'a> SnapshotView<'a> {
    pub fn blocks(&self) -> &[BlockSnapshot];
    pub fn block_at_point(&self, p: Point) -> Option<&BlockSnapshot>;
}
```

### 7.2. События ядра

В `otty-libterm::terminal::TerminalEvent` добавляется:

```rust
enum TerminalEvent {
    // ...
    BlocksChanged { blocks: Arc<Vec<BlockMetaPublic>> },
}
```

Использование:

- При изменении списка/метаданных блоков ядро отправляет `BlocksChanged`.
- Фронтенд может использовать это как API для работы с блоками, не завися от snapshot.

## 8. UI‑поведение в `otty-ui/terminal`

### 8.1. Отображение блоков

В `otty-ui/terminal/src/view.rs::TerminalView::draw`:

- Из `SnapshotView` берётся:
  - список клеток (`cells`), как сейчас;
  - список `BlockSnapshot` (`view.blocks()`).
- При отрисовке:
  - для каждого блока вычисляется прямоугольник:
    - `y_start`/`y_end` по `start_line` и `line_count` с учётом `display_offset`;
    - `x`/`width` охватывают всю ширину терминала.
  - если блок является:
    - hovered — рисуется рамка/обводка;
    - selected — может использоваться более яркая рамка.

Отступы между блоками не добавляются, блоки идут вплотную; рамка визуально отделяет их.

### 8.2. Состояние и взаимодействия

`TerminalViewState` дополняется:

```rust
pub(crate) struct TerminalViewState {
    // ...
    pub hovered_block_id: Option<String>,
    pub selected_block_id: Option<String>,
}
```

Поведение:

- наведение: при движении мыши определяется блок через `block_at_point`, состояние `hovered_block_id` обновляется, view перерисовывается, временно подсвечивая рамку вокруг блока;
- клик: при клике ЛКМ `selected_block_id` устанавливается в `hovered_block_id`, рамка закрепляется и блок становится «активным» для взаимодействия;
- активный блок с типом `Command`/`FullScreen` предоставляет стандартные действия с блоком (например, копирование вывода);
- активный блок с типом `Prompt` в будущем будет использоваться как область ввода (input), доступная для редактирования и отправки команд.

### 8.3. Простое взаимодействие (копирование блока)

В `otty-ui/terminal/src/input.rs` / `InputManager`:

- вводится новое действие, например `BindingAction::CopyBlock`;
- по умолчанию его можно повесить на хоткей (например, `Cmd+Shift+C`) или отдельную команду;
- при выполнении `CopyBlock`:
  - определяется `selected_block_id`;
  - через `SnapshotView` и `BlockSnapshot` собирается текст всех клеток блока (аналогично уже существующему `selectable_content`);
  - результат пишется в clipboard.

## 9. Shell‑поддержка

На стороне оболочки (bash/zsh/fish) реализуются плагины/скрипты, которые:

- подписываются на хуки:
  - `preexec` — перед выполнением команды;
  - `precmd` — перед показом нового промпта;
  - в bash — через `bash-preexec` или аналогичный механизм;
- формируют JSON по описанному формату, соблюдая лимиты длины;
- отправляют DCS‑строки в stdout:

```text
printf '\033P'        # ESC P
printf 'otty-block;%s' "$json"
printf '\033\\'       # ESC \
```

Поддерживаемые на старте оболочки: **bash**, **zsh**, **fish**.

## 10. Открытые вопросы и будущая работа

- Политика очистки старых блоков при `max_blocks`:
  - удалять только полностью завершённые блоки;
  - сделать поведение настраиваемым пользователем.
- Расширения API:
  - навигация по блокам (jump next/prev);
  - поиск/фильтрация по блокам;
  - отображение блока в отдельной панели.
