# 08. Поведение промпт‑блоков и подготовка к input‑редактору

## Обзор

Восьмая задача: явно отделить промпт‑блоки (`BlockKind::Prompt`) от командных и полноэкранных, а также подготовить их к будущему использованию как области ввода. На этом шаге не реализуется полноценный input‑editor, только разграничение типов и базовое визуальное/поведенческое отличие.

См. спецификацию: `specs/spec.md`, разделы 4 и 8.2.

## Что уже есть

- Задачи 01–07:
  - `BlockKind` определён и используется в `BlockMeta`/`BlockSnapshot`.
  - `BlockSurface` создаёт/завершает блоки по `BlockEvent`.
  - UI умеет выбирать блоки и копировать их содержимое.

## Что нужно сделать

1. Классификация блоков в `BlockSurface`:
   - Для `BlockPhase::Preexec`:
     - блоки создаются с `kind = Command`.
   - Для `BlockPhase::Precmd`:
     - если активный командный блок ещё не завершён — пометить его как завершённый;
     - создать новый блок с `kind = Prompt` (под будущий ввод).
   - Для предприятий в full‑screen режиме:
     - по необходимости использовать `kind = FullScreen` (если ALT_SCREEN определяется).

2. Экспорт `BlockKind` в `BlockMetaPublic`/`BlockSnapshot`:
   - Убедиться, что поле `kind: BlockKind` корректно заполняется при построении snapshot’а.

3. В UI:
   - В `TerminalView::draw`:
     - для блоков `Prompt` можно использовать слегка отличающийся стиль рамки (например, другой оттенок или пунктир) — по желанию;
   - В логике `CopyBlock`:
     - определить поведение для `Prompt`:
       - либо копировать текст промпта как обычный блок;
       - либо временно игнорировать (например, если `kind == Prompt`, ничего не делать).

4. Подготовка к input‑редактору:
   - В `InputManager`/`TerminalViewState` предусмотреть доступ к типу активного блока:
     - простая helper‑функция на базе `SnapshotView` и `selected_block_id`.
   - Это потребуется в будущей задаче по внедрению отдельного редактора ввода в промпт‑блоках.

## Ожидаемый результат

- Блоки имеют корректный `BlockKind`, отличающий промпт, команды и full‑screen.
- UI способен, при необходимости, визуально отличать промпт‑блоки от остальных.
- Логика копирования/выбора блоков знает о типе блока и может варьировать поведение.

## Как протестировать

1. Юнит‑тесты `BlockSurface`:
   - Сценарий `Preexec` → `Exit` → `Precmd` должен приводить к:
     - одному командному блоку и одному промпт‑блоку;
     - корректным `kind` в обоих.

2. Юнит‑тесты UI:
   - Создать snapshot с блоками разных типов;
   - проверить, что helper‑функция «тип активного блока» возвращает ожидаемые значения.

3. Ручной тест:
   - Включить shell‑хуки `preexec`/`precmd`;
   - убедиться, что после нескольких команд и переходов к новому промпту появляются соответствующие блоки и промпт‑блок визуально отличим (если добавлен стиль).
