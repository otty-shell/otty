# План реализации блоковой модели терминала (Warp‑подобные блоки)

Этот документ описывает поэтапную реализацию RFC из `specs/spec.md` для текущего кода OTTY. Каждый шаг должен завершаться состоянием, которое можно собрать и протестировать, не ломая существующее поведение.

---

## 0. Анализ текущего состояния

Что уже есть:

- `otty-libterm`:
  - `TerminalEngine<P, E, S>` обобщён по `SurfaceActor + SurfaceModel` — позволяет подменить модель поверхности на `BlockSurface`.
  - `TerminalSurfaceActor` уже является адаптером `Action` → `SurfaceActor` и точкой, где удобно обработать новые действия (`Action::BlockEvent`).
  - `SnapshotOwned`/`SnapshotView` уже содержат плоский список клеток, состояние курсора, режимы, damage и т.п.
- `otty-surface`:
  - есть базовый `Surface`, грид, snapshot‑механизм и привязка к `SurfaceActor`.
  - уже реализовано выделение, скролл, поиск, гиперссылки.
- `otty-ui/terminal`:
  - `Engine` уже умеет принимать `SnapshotOwned` и отдаёт их в `TerminalView`.
  - `TerminalView` рисует плоский грид, поддерживает selection и hover по hyperlink’ам.
  - `InputManager` уже реализует мышь/клавиатуру, копирование выделения, скролл и т.п.

Чего нет (на момент изначального плана):

- В `otty-escape` нет DCS‑протокола блоков и `Action::BlockEvent` (выполнено в задаче 01).
- В `otty-surface` блоковая модель уже заведена (`BlockSurface`, `BlockMeta`,
  обработка `Action::BlockEvent`), но snapshot по‑прежнему экспортирует только
  активный блок, а публичных структур (`BlockSnapshot`/`BlockMetaPublic`) ещё нет.
  Поля `cmd`/`cwd`/`shell` от DCS при этом теряются.
- В `TerminalEvent` нет событий о блоках, `SnapshotOwned` не содержит информации о блоках.
- В UI нет выделения/рамок по блокам, нет понятия активного блока и операций над ним.

---

## 1. Базовая проводка DCS → Action (метаданные блоков без влияния на вывод)

Цель: добавить протокол блоков в `otty-escape` и провести его до `TerminalSurfaceActor`, не меняя модель поверхности и отрисовку.

Шаги:

1.1. В `otty-escape`:

- Добавить типы:
  - `BlockKind` (Command / Prompt / FullScreen).
  - `BlockPhase` (Preexec / Exit / Precmd).
  - `BlockMeta` (id, kind, cmd, cwd, started_at, finished_at, exit_code, shell, is_alt_screen).
- Добавить новый вариант `Action::BlockEvent(BlockEvent)` с `BlockEvent { phase, meta }`.

1.2. В DCS‑парсере `otty-escape`:

- Распознать DCS с префиксом `otty-block;`.
- Выделить JSON‑строку, распарсить её в `BlockMeta`/`BlockPhase`:
  - учесть лимиты длины (`cmd`, `cwd`, общий JSON);
  - на любые ошибки парсинга — логировать и игнорировать DCS.

1.3. В `otty-libterm::terminal::surface_actor`:

- Добавить обработку `Action::BlockEvent(..)`:
  - пока только логировать событие и не изменять `Surface` (без `BlockSurface`).

Проверка:

- Собрать workspace (`cargo build --workspace`).
- Запустить терминал без shell‑хуков — поведение не меняется.
- (Опционально) сымитировать DCS с тестовой строки (через PTY или юнит‑тест в `otty-escape`) и убедиться, что `Action::BlockEvent` вызывается и не ломает отрисовку.

## 2. Введение много‑гридовой модели `BlockSurface` в otty‑surface

Цель: реализовать в `otty-surface` модель `BlockSurface`, которая внутри хранит несколько `Surface` — по одному на блок, и реализует `SurfaceActor + SurfaceModel`, но пока без snapshot блоков наружу.

Шаги:

2.1. Добавить новые структуры в `otty-surface`:

- `BlockKind` и `BlockMeta` (внутренние, концептуально эквивалентные типам из `otty-escape`, но не зависящие от него напрямую).
- `Block` с полями `{ meta: BlockMeta, surface: Surface }`.
- `BlockSurface { blocks: Vec<Block>, active: usize, max_blocks: usize }`.
  - `BlockMeta` минимум: `id`, `kind`, `started_at`, `finished_at`, `exit_code`, `is_alt_screen`.
  - В будущем к этому типу будет маппиться метадата из `Action::BlockEvent`.

2.2. Реализовать `SurfaceActor` для `BlockSurface`:

- Для всех операций (`print`, `scroll`, `clear`, `set_mode`, и т.д.) делегировать их в текущий активный блок: `self.blocks[self.active].surface`.
- На этом шаге **не** менять активный блок и **не** реагировать на `Action::BlockEvent` — блоки пока не создаются/завершаются автоматически.
- Добавить минимальные внутренние методы для управления блоками (создание, выбор активного, завершение), но оставить их неиспользованными до задачи 3.

2.3. Реализовать `SurfaceModel` для `BlockSurface`:

- `snapshot_owned` на этом этапе:
  - возвращает snapshot **только активного** блока (как текущий `Surface`), чтобы не менять поведение до внедрения блоковой отрисовки;
  - собирает damage/selection/режимы только из активного `Surface`.

2.4. Подключить `BlockSurface` в `otty-libterm`:

- В `TerminalBuilder` (тип `DefaultSurface`) заменить базовый `Surface` на `BlockSurface` (оборачивая один `Surface` внутри).
- Убедиться, что без каких‑либо блоковых событий поведение идентично текущему (один блок = один `Surface`).

Проверка:

- `cargo test -p otty-surface` и `cargo test -p otty-libterm`.
- Ручной запуск приложения: вывод должен рендериться как и раньше, хотя теперь внутри используется `BlockSurface` с одним блоком.

## 3. Связка `Action::BlockEvent` ↔ `BlockSurface` (без UI‑изменений)

Цель: научить ядро создавать/завершать блоки по DCS‑событиям, но пока рендерить только активный блок (без визуальной блоковой разметки).

Шаги:

3.1. В `BlockSurface`:

- Добавить методы:
  - `begin_command_block(meta: BlockMeta)` — создание нового блока с типом `Command` и новым `Surface`;
  - `end_block(id)` — пометка блока как завершённого (`finished_at`, `exit_code`);
  - аналогичные операции для промпт‑блока и full‑screen блока при необходимости.
- Реализовать политику `max_blocks`:
  - при превышении `max_blocks` удалять самый старый завершённый блок (не трогая незавершённые).

3.2. В `TerminalSurfaceActor`:

- Для `Action::BlockEvent { phase: Preexec, meta }`:
  - вызвать `BlockSurface::begin_command_block(meta)` (или подходящий метод);
  - переключить активный блок на вновь созданный.
- Для `Action::BlockEvent { phase: Exit, meta }`:
  - вызвать `BlockSurface::end_block(meta.id)`.
- Для `Action::BlockEvent { phase: Precmd, meta }`:
  - пометить предыдущий командный блок завершённым (если ещё не завершён);
  - при необходимости создать промпт‑блок.

3.3. Определить `is_alt_screen`:

- При создании блока (на `Preexec`) запомнить состояние `SurfaceMode` (ALT_SCREEN) и выставить `meta.is_alt_screen` в зависимости от режима.

3.4. На этом шаге `snapshot_owned` всё ещё может возвращать только активный блок (визуально поведение прежнее).

Проверка:

- Юнит‑тесты в `otty-libterm`:
  - с помощью `StubParser`, генерирующего `Action::BlockEvent` + простые `Print`, проверить, что `BlockSurface` создаёт несколько блоков и корректно переключает активный.
- Ручной тест с прототипом shell‑хука: запустить терминал, убедиться, что визуальное поведение не ломается.

## 4. Snapshot блоков и экспорт в `SnapshotOwned` / `SnapshotView`

Цель: расширить snapshot так, чтобы он содержал сведения обо всех блоках, и «сшить» их в один плоский viewport для UI.

Шаги:

4.0. Расширить внутренние метаданные блоков:

- В `otty-surface/src/block.rs` добавить к `BlockMeta` поля `cmd: Option<String>`,
  `cwd: Option<String>` и `shell: Option<String>` (эти значения уже приходят из
  `otty-escape::BlockMeta`, но сейчас отбрасываются).
- В `BlockSurface::begin_block` и `end_block_by_id` аккуратно обновлять
  метаданные: новые значения переносят только `Some(..)`, чтобы `cmd`/`cwd`
  из `Preexec` не затирались `None` из `Exit`.
- В `TerminalSurfaceActor::handle_block_event` перестать терять метаданные
  (`cmd`, `cwd`, `shell`, `started_at`, `finished_at`); передавать всё в
  `BlockSurface`, даже если конкретная реализация `SurfaceActor` игнорирует блоки.
- Обновить существующие тесты `block_surface_*` и/или добавить новые, которые
  проверяют, что `cmd`/`cwd`/`shell` сохраняются после `begin_block`/`end_block`.

4.1. В `otty-surface::snapshot`:

- Для `BlockSurface::snapshot_owned`:
  - пройти по всем блокам в порядке истории;
  - для каждого блока:
    - получить snapshot вложенного `Surface`;
    - сместить `start_line` с учётом накопленной высоты предыдущих блоков;
    - собрать `BlockSnapshot { id, kind, meta, start_line, line_count, is_alt_screen }`.
  - собрать общий плоский список клеток `SnapshotCell` по всем блокам подряд.
- При сборке `SnapshotCell` сдвигать `point.line` на накопленный offset, чтобы
  глобальные координаты соответствовали общему viewport’у.

4.2. Расширить `SnapshotOwned`/`SnapshotView`:

- Добавить поле `blocks: Vec<BlockSnapshot>`.
- В `SnapshotView` реализовать:
  - `fn blocks(&self) -> &[BlockSnapshot]`;
  - `fn block_at_point(&self, p: Point) -> Option<&BlockSnapshot>`.

4.3. Убедиться, что режимы scrollback и `display_offset` работают ожидаемо:

- при большом числе строк история блоков должна прокручиваться как единый поток.
- `BlockSurface::scroll_display` больше не может просто делегировать активному
  блоку — требуется общее `display_offset`, которое ходит по суммарной высоте
  всех блоков. Отдельные `Surface` внутри блоков продолжают вести свой
  `display_offset`, но глобальный offset должен вычислять, в какой блок и на
  какую относительную позицию попадает текущий viewport.

Проверка:

- Новый модуль тестов в `otty-surface`:
  - несколько блоков с разной высотой; проверить корректность `start_line`/`line_count`.
- Ручной запуск:
  - без изменений UI визуально всё ещё грид, но логирование по debug может показывать наличие нескольких блоков в snapshot.

## 5. Прокидывание информации о блоках до UI (`otty-ui/terminal`)

Цель: сделать блоки доступны на уровне UI, но пока без визуальных рамок.

Шаги:

5.1. В `otty-ui/terminal/src/engine.rs`:

- Ничего менять не требуется: `Engine` уже хранит `Arc<SnapshotOwned>`. После расширения snapshot блоки будут доступны через `self.snapshot.view().blocks()`.

5.2. В `otty-ui/terminal/src/term.rs`:

- При необходимости добавить геттеры:
  - `pub fn blocks(&self) -> &[BlockSnapshot]`, делегирующий к `self.engine.snapshot().view().blocks()`.
  - Это нужно для дальнейшего внешнего API, но непосредственно отрисовку можно строить прямо из `view` в `TerminalView`.

Проверка:

- Сборка `otty-ui/terminal`.
- Простая проверка: логировать количество блоков при каждом `ContentSync` и убедиться, что они появляются при отправке DCS.

## 6. Визуальная отрисовка рамок блоков в `TerminalView`

Цель: визуально отделить блоки в UI, ввести hover/selected состояние и закрепление активного блока по клику, без изменения остального поведения ввода.

Шаги:

6.1. В `TerminalViewState` (`otty-ui/terminal/src/view.rs`):

- Добавить поля:
  - `hovered_block_id: Option<String>`;
  - `selected_block_id: Option<String>`.

6.2. В `TerminalView::draw`:

- Получить `let view = content.view();`.
- На основе `view.blocks()`:
  - для каждого блока вычислить геометрию прямоугольника:
    - `y` по `start_line`, `line_count`, `display_offset` и `cell_height`;
    - `x` = начало layout, `width` = ширина layout.
- Перед отрисовкой текста:
  - для блока, чей `id` равен `selected_block_id`, нарисовать рамку более ярким цветом;
  - для блока, чей `id` равен `hovered_block_id` (и не равен `selected_block_id`), нарисовать рамку менее ярким цветом.

6.3. Обновление hover/selected:

- При движении мыши:
  - уже вычисляется `mouse_position_on_grid` в `InputManager`;
  - добавить вызов `block_at_point` и обновление `hovered_block_id` в `TerminalViewState`.
- При клике ЛКМ:
  - если `hovered_block_id` есть, присвоить её в `selected_block_id`.

6.4. Не менять существующую логику selection, scroll, hyperlink hover.

Проверка:

- Ручной запуск:
  - при наведении мыши рамка появляется над соответствующим блоком;
  - при клике рамка остаётся (selected), hover меняется при наведении на другие блоки.

## 7. Минимальное взаимодействие с блоками (копирование активного блока)

Цель: реализовать простое действие над активным блоком — копирование его содержимого в буфер обмена.

Шаги:

7.1. В `InputManager` (`otty-ui/terminal/src/input.rs`):

- Добавить новый `BindingAction::CopyBlock`.
- Определить дефолтный биндинг (например, `Cmd+Shift+C` либо отдельная комбинация, не конфликтующая с текущим `Copy`).

7.2. Обработка `CopyBlock`:

- Получить текущий `SnapshotArc` и `SnapshotView`.
- По `TerminalViewState.selected_block_id` найти `BlockSnapshot`.
- Собрать текст блока:
  - пройти по `view.cells`, выбрать клетки, попадающие в диапазон линий блока;
  - аккуратно вставить переводы строк по границе линий.
- Записать текст в `clipboard`.

7.3. Для блоков с типом `Prompt`:

- На этой итерации можно вести себя так же, как для командных блоков (копировать вывод), либо временно игнорировать (в дальнейшем заменить на ввод).

Проверка:

- Ручной запуск:
  - выбрать блок кликом;
  - нажать биндинг `CopyBlock`;
  - убедиться, что в буфере обмена текст именно этого блока.

## 8. Поведение промпт‑блоков (подготовка к input‑редактору)

Цель: отделить промпт‑блоки по типу и подготовить их к будущему использованию как области ввода, не реализуя сам редактор.

Шаги:

8.1. Классификация блоков:

- На уровне `BlockMeta.kind`:
  - `Command` — блоки, созданные на `preexec`.
  - `Prompt` — блоки, созданные на `precmd` (если решено явно их создавать).
  - `FullScreen` — блоки, где `is_alt_screen = true`.

8.2. В UI:

- При отрисовке можно:
  - для блоков `Prompt` использовать другой стиль рамки (например, другой цвет);
  - в логике `CopyBlock` (при желании) пока либо копировать вывод промпта, либо пропускать такие блоки.

8.3. Подготовить API‑крючки:

- В `TerminalViewState`/`InputManager` предусмотреть возможность:
  - проверять `BlockKind` выбранного блока;
  - различать “активный промпт” и “активную команду” в будущих действиях.

Проверка:

- Ручной: при включённых хуках promt/command блоки должны различаться по типу и стилю рамки (если стили будут добавлены).

## 9. Итоговая проверка и уборка

Шаги:

9.1. Тесты:

- `cargo test --workspace` — все тесты должны проходить.
- При необходимости добавить:
  - тесты в `otty-escape` на корректный парсинг JSON в `BlockEvent`;
  - тесты в `otty-surface` на формирование `BlockSnapshot`;
  - тесты в `otty-ui/terminal` на поведение `CopyBlock` и выбор блоков.

9.2. Ручные сценарии:

- Сессия без shell‑хуков — терминал работает как раньше (один блок).
- Сессия с хуками:
  - несколько команд подряд → несколько блоков, рамка по hover/selected;
  - команды без перевода строки (`printf`) → блоки формируются по `preexec/exit`, визуально не ломают историю;
  - запуск `vim`/`htop` (ALT_SCREEN) → один full‑screen блок.

9.3. Документация:

- Обновить `specs/spec.md` при необходимости, если что‑то будет уточнено в ходе имплементации.
- Добавить краткое описание включения блоков и shell‑хуков в `README.md`/доках UI (отдельной задачей).

На этом этапе функциональность блоков считается минимально реализованной и пригодной для дальнейшего развития (input‑редактор для промпт‑блоков, навигация по блокам, поиск и т.п.).
