# 05. Прокидывание информации о блоках до UI

## Обзор

Пятая задача: сделать данные о блоках доступными на уровне `otty-ui/terminal`, не меняя пока визуальное представление. UI должен иметь возможность получить список блоков и их метаданные из последнего snapshot’а.

См. спецификацию: `specs/spec.md`, разделы 7.1 и 7.2 (частично).

## Что уже есть

- Задачи 01–04:
  - `SnapshotOwned` содержит `blocks: Vec<BlockSnapshot>`.
  - `SnapshotView` предоставляет `blocks()` и `block_at_point`.
- `otty-ui/terminal`:
  - `Engine` хранит `snapshot: Arc<SnapshotOwned>` и обновляет его по событиям `ContentSync`.
  - `Terminal` инкапсулирует `Engine` и отдаёт snapshot через внутреннее API (`engine.snapshot()`).
  - `TerminalView` уже использует `snapshot.view()` для отрисовки клеток.

## Что нужно сделать

1. Сделать данные snapshot’а и блоков доступными из публичной API `Terminal` (`otty-ui/terminal/src/term.rs`):
   - добавить `pub fn snapshot_arc(&self) -> SnapshotArc`, просто делегирующую к `self.engine.snapshot()`; это понадобится тестам и будущим UI-компонентам вне `view.rs`;
   - добавить `pub fn blocks(&self) -> Vec<BlockSnapshot>` (или `Arc<[BlockSnapshot]>`, если понадобится избежать копирования) и сформировать её из `self.engine.snapshot().view().blocks().to_vec()`;
   - документировать оба метода так, чтобы было понятно, что они возвращают данные последнего `ContentSync`.

2. После добавления геттеров проверить, не нужны ли дополнительные re-export’ы:
   - при необходимости переэкспортировать `SnapshotArc`/`BlockSnapshot` из `lib.rs`, чтобы потребителям `Terminal` не приходилось тянуть их из `otty-libterm` вручную.

3. (Проверка реализации, правок в коде не требуется): убедиться, что `Event::ContentSync { frame, .. }` в `Terminal::handle_event` продолжает звать `self.engine.sync_snapshot(frame)` — этим шагом данные блоков попадают в `snapshot_arc()`.

4. Логирование количества блоков остаётся опциональным; если оно не нужно, просто обновите комментарии в коде/описании, что дебаг можно включить при необходимости.

## Ожидаемый результат

- Код UI имеет доступ к списку блоков из snapshot’а.
- Внешние потребители `otty-ui/terminal` (если есть) могут, при необходимости, запросить информацию о блоках через `Terminal`.
- Визуальное представление терминала пока не меняется.

## Как протестировать

1. Юнит‑/интеграционные тесты:
   - в `otty-ui/terminal` добавить модульный тест, который:
     - создаёт `Terminal` с минимальными тестовыми настройками (можно мокнуть `engine` через прямую инициализацию/внедрение тестового `SnapshotOwned`);
     - вручную вызывает `terminal.engine.sync_snapshot(test_frame)` и затем проверяет, что `terminal.blocks()` возвращает нужные `BlockSnapshot` (как минимум сравнить `id`/`kind`/`line_count`);
     - аналогично проверить `snapshot_arc()` — метод должен отдавать тот же `Arc<SnapshotOwned>` без копирования.

2. Ручной тест:
   - запустить UI с включёнными shell‑хуками блоков;
   - через REPL/лог временно вызвать новые геттеры и убедиться, что список блоков обновляется после каждой команды (дополнительно можно вывести количество блоков в лог для контроля).
