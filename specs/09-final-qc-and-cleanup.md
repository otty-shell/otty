# 09. Итоговая проверка и уборка

## Обзор

Финальная задача: удостовериться, что вся цепочка блоков (протокол → ядро → snapshot → UI) работает стабильно, завершить тестирование и минимальную документацию, а также убрать временное логирование/отладочные конструкции.

См. спецификацию: `specs/spec.md`, раздел 9.

## Что уже есть

- Задачи 01–08:
  - DCS‑протокол блоков реализован и генерирует `Action::BlockEvent`.
  - `BlockSurface` хранит историю блоков и переключает активный.
  - Snapshot экспортирует `BlockSnapshot` и даёт доступ к блокам через `SnapshotView`.
  - UI визуализирует блоки, позволяет их выбирать и копировать содержимое.

## Что нужно сделать

1. Прогнать тесты:
   - `cargo test --workspace`.
   - При необходимости добавить недостающие тесты:
     - на граничные сценарии (много блоков, лимит `max_blocks`, длинные команды).

2. Ручное тестирование ключевых сценариев:
   - Без shell‑хуков:
     - терминал ведёт себя как обычный; блоки либо не создаются, либо есть один «дефолтный».
   - С включёнными хуками:
     - несколько простых команд (`ls`, `echo`, `printf` без `\n`);
     - команды с ошибками (не нулевой exit‑код);
     - полноэкранные приложения (`vim`, `htop`), если доступно:
       - должен появляться отдельный full‑screen блок.
   - Проверка рамок:
     - hover и selected, отсутствие артефактов при скролле.
   - Проверка `CopyBlock`:
     - корректность копируемого текста, отсутствие соседнего вывода.

3. Уборка:
   - Удалить/ослабить избыточное временное логирование (debug‑выводы, оставленные при разработке).
   - Проверить наличие `TODO`/`FIXME`, связанных с этим RFC, и либо зарегистрировать отдельные задачи, либо закрыть их.

4. Документация:
   - Актуализировать `specs/spec.md`, если какие‑то детали реализации отличились от первоначального RFC (минимальными правками).
   - При необходимости добавить краткий раздел в README или отдельную doc‑страницу:
     - как включить блоки (shell‑хуки);
     - какие возможности доступны сейчас (визуальное разделение, копирование блока).

## Ожидаемый результат

- Все тесты проекта проходят.
- Блоковая модель работает в основных сценариях, не ломая базовое терминальное поведение.
- Репозиторий не содержит временного дебаг‑кода, а документация соответствует реализованному поведению.
