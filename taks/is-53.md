# IS-53. Совместимый DEB-пакет для Ubuntu 20.04, 22.04 и 24.04

## Контекст

Текущий DEB-пакет `otty_0.1.0_amd64.deb` собирается в GitHub Actions на
`ubuntu-latest`. На момент возникновения задачи это окружение использует более
новые системные библиотеки, чем Ubuntu 20.04 и Ubuntu 22.04. В результате в
метаданные пакета попадают зависимости, недоступные на старых версиях Ubuntu:

```text
otty depends on libc6 (>= 2.39)
otty depends on libssl3t64 (>= 3.0.0)
```

На Ubuntu 20.04 установлена glibc 2.31, а пакет `libssl3t64` относится к более
новым дистрибутивам. Поэтому существующий пакет невозможно установить на
Ubuntu 20.04.

Проблема состоит из двух независимых частей:

1. `ssh2`/`libssh2-sys` динамически связывается с системными `libssl` и
   `libcrypto`, найденными в build environment.
2. Бинарник собирается с использованием glibc из `ubuntu-latest`, поэтому
   требует версию glibc не ниже версии build environment.

В `otty-pty` используется `ssh2 = "0.9.5"`. Эта версия предоставляет feature
`vendored-openssl`, который передает `vendored-openssl` в `libssh2-sys`, а тот
включает `openssl-sys/vendored`. Это позволяет собрать OpenSSL из исходников и
статически включить его в итоговый бинарник. Однако этот feature сам по себе не
понижает требуемую версию glibc: для этого весь Linux-бинарник необходимо
собирать в окружении самой старой поддерживаемой Ubuntu.

## Цель

Подготовить один DEB-пакет `amd64`, который:

- устанавливается на Ubuntu 20.04, Ubuntu 22.04 и Ubuntu 24.04;
- не зависит от системных пакетов `libssl`, `libcrypto` или `libssh2`;
- требует не более `libc6 (>= 2.31)`;
- продолжает поддерживать SSH-функциональность OTTY;
- автоматически собирается и проверяется при публикации релизного тега.

Отдельные DEB-пакеты для Ubuntu 20.04 и 22.04 создавать не требуется, если один
пакет, собранный на Ubuntu 20.04, успешно проходит всю матрицу совместимости.

## Текущее состояние

- Зависимость `ssh2` объявлена в `otty-pty/Cargo.toml` без feature
  `vendored-openssl`.
- `libssh2-sys` уже собирает bundled `libssh2` по умолчанию, но OpenSSL на Unix
  остается системной динамической зависимостью.
- Linux-пакеты DEB и RPM собираются одним job в
  `.github/workflows/build-packages.yml` на `ubuntu-latest`.
- `cargo-deb` автоматически формирует зависимости пакета на основании
  динамических зависимостей итогового ELF-бинарника.
- В `package.metadata.deb` не следует вручную подменять `depends`, чтобы скрыть
  реально существующие динамические зависимости.

## Что нужно сделать

### 1. Включить vendored OpenSSL для SSH backend

Изменить зависимость `ssh2` в `otty-pty/Cargo.toml`:

```toml
ssh2 = { version = "0.9.5", features = ["vendored-openssl"] }
```

После изменения необходимо обновить `Cargo.lock`. В lockfile должна появиться
зависимость, через которую `openssl-sys` собирает vendored OpenSSL, например
`openssl-src` для текущей версии dependency graph.

Требования к результату:

- bundled `libssh2` остается статически связан с приложением;
- OpenSSL и libcrypto собираются из исходников и статически связываются с
  приложением;
- ELF-бинарник не содержит `DT_NEEDED` для `libssh2.so`, `libssl.so` и
  `libcrypto.so`;
- SSH-соединение и поддерживаемые способы аутентификации продолжают работать.

Не следует копировать `.so`-файлы внутрь DEB-пакета и настраивать `RPATH` как
основное решение. Для используемого dependency graph предпочтительна
статическая vendored-сборка через Cargo feature.

### 2. Собирать DEB в Ubuntu 20.04 build environment

Изменить release workflow так, чтобы DEB-бинарник и сам DEB-пакет создавались
внутри Ubuntu 20.04 container/build image с glibc 2.31.

Рекомендуемая схема:

- GitHub Actions runner может оставаться `ubuntu-latest`;
- DEB job должен выполняться в `ubuntu:20.04` либо использовать отдельный
  Dockerfile на базе `ubuntu:20.04`;
- установка Rust toolchain, `cargo-deb`, native build dependencies, сборка
  `otty` и запуск `cargo deb` должны происходить внутри этого окружения;
- использовать актуальный Rust toolchain, заданный проектом, а не системный
  пакет Rust из Ubuntu;
- build image должен содержать минимально необходимые инструменты для сборки
  Rust/C-кода и vendored OpenSSL, включая C toolchain, `make`, `perl`,
  `pkg-config`, сертификаты и Debian packaging tools;
- список системных build dependencies должен быть явным и воспроизводимым;
- generated package и бинарник не должны повторно линковаться или
  пересобираться на host runner после выхода из Ubuntu 20.04 environment.

Сборка на новой системе с последующим ручным изменением поля `Depends` не
допускается: такой пакет останется ABI-несовместимым, даже если `dpkg` позволит
его установить.

### 3. Разделить DEB- и RPM-сборки

Сейчас RPM и DEB создаются в одном Linux job. Нужно разделить их на независимые
jobs либо иначе обеспечить, что смена окружения DEB-сборки не изменит RPM
artifact.

Ожидаемое поведение:

- DEB собирается в Ubuntu 20.04 environment;
- существующая RPM-сборка продолжает работать в своем подходящем окружении;
- оба artifacts загружаются в GitHub Actions с однозначными именами;
- падение проверки совместимости DEB блокирует публикацию неисправного DEB.

### 4. Добавить матрицу smoke-тестов установки

Собранный один раз DEB-пакет нужно проверить без пересборки как минимум в
следующих чистых окружениях:

- Ubuntu 20.04;
- Ubuntu 22.04;
- Ubuntu 24.04.

В каждом окружении необходимо:

1. Установить runtime-зависимости стандартным пакетным менеджером.
2. Установить именно один и тот же DEB artifact.
3. Проверить, что `dpkg` не сообщает о неудовлетворенных зависимостях.
4. Проверить `ldd /usr/bin/otty` на отсутствующие библиотеки.
5. Проверить, что бинарник может быть загружен процессом без ошибки loader-а.

Если полноценный запуск GUI требует display server, для smoke-теста следует
использовать Xvfb/подходящий headless backend либо добавить минимальный режим,
который проверяет загрузку приложения без открытия окна. Ошибка вида “display
not available” допустима только после того, как подтверждено отсутствие ошибки
динамического loader-а; она не заменяет проверку запуска приложения.

### 5. Проверить SSH-функциональность

Vendored-сборка меняет криптографический backend, поэтому одной проверки
установки недостаточно. Нужен тест либо задокументированный reproducible smoke
scenario, подтверждающий:

- создание SSH session через существующий `otty-pty` backend;
- handshake с тестовым SSH server;
- аутентификацию как минимум одним поддерживаемым способом;
- открытие PTY channel;
- обмен данными и корректное закрытие session.

Тест не должен обращаться к внешнему публичному SSH server. Следует запускать
локальный ephemeral test server/container с фиксированной тестовой
конфигурацией и непроизводственными credentials.

### 6. Обновить документацию релизов

В документации Linux installation/release artifacts указать:

- минимально поддерживаемую версию Ubuntu — 20.04;
- один DEB artifact предназначен для Ubuntu 20.04, 22.04 и 24.04 на `amd64`;
- OpenSSL включен в приложение статически для переносимости;
- обновление OTTY до свежей версии также обновляет встроенный OpenSSL.

Не следует обещать совместимость с произвольным Linux-дистрибутивом только на
основании успешной проверки трех версий Ubuntu.

## Security и сопровождение

Статически включенный OpenSSL не получает security fixes через обновление
системного пакета `libssl`. Поэтому необходимо:

- сохранять `Cargo.lock` в репозитории и отслеживать обновления
  `openssl-sys`/`openssl-src`;
- пересобирать и перевыпускать OTTY при security update vendored OpenSSL;
- не кэшировать готовый OpenSSL между релизами так, чтобы cache мог скрыть
  обновление dependency graph;
- проверить лицензии новых transitive dependencies через `cargo deny check`;
- не отключать криптографические проверки или проверку host key ради
  прохождения smoke-теста.

## Вне scope

- Flatpak packaging.
- AppImage packaging.
- Полностью статическая musl-сборка GUI-приложения.
- Поддержка Ubuntu старше 20.04.
- Поддержка архитектур, отличных от `amd64`.
- Ручное добавление `libssl*.so` в DEB.
- Изменение SSH API или пользовательских SSH-сценариев, не требуемое для
  vendored linkage.

AppImage можно рассмотреть отдельной задачей, но он не устраняет зависимость от
glibc автоматически: AppImage, собранный на Ubuntu 24.04, также может требовать
новую glibc. Flatpak с зафиксированным runtime решает эту задачу иначе и также
должен проектироваться отдельно.

## Обязательные проверки

После реализации выполнить стандартный quality gate проекта:

```bash
cargo +nightly fmt
cargo clippy --workspace --all-targets --all-features -- -D warnings
cargo deny check
cargo test --workspace --all-features
cargo llvm-cov --workspace --all-features --fail-under-lines 80
```

Coverage измененного кода не должен снижаться; общий baseline остается не ниже
80% строк.

## Acceptance criteria

- В `otty-pty` включен feature `ssh2/vendored-openssl`.
- `Cargo.lock` зафиксировал vendored OpenSSL dependency graph.
- DEB-бинарник полностью собирается внутри Ubuntu 20.04 build environment.
- DEB и RPM jobs не влияют на build environment друг друга.
- Один и тот же DEB artifact без пересборки успешно устанавливается на Ubuntu
  20.04, 22.04 и 24.04.
- Во всех трех окружениях `ldd` не показывает `not found`, а бинарник проходит
  loader/startup smoke-test.
- SSH handshake, authentication, PTY open и обмен данными проходят с vendored
  OpenSSL backend.
- Документация указывает фактический диапазон поддерживаемых Ubuntu и
  ограничение по архитектуре.
- Форматирование, clippy, deny, workspace tests и coverage gate проходят без
  ошибок и предупреждений.

## Результат задачи

Релизный pipeline публикует один проверенный `amd64` DEB-пакет, совместимый с
Ubuntu 20.04, Ubuntu 22.04 и Ubuntu 24.04, без runtime-зависимости от системного
OpenSSL и без требования glibc новее 2.31.
